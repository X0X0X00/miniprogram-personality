**日期：** 2026-04-13  
**范围：** 在现有微信小程序基础上新增双速模式 + 情侣契合模式

## 一、整体架构

### 三种模式

| 模式 | 题目数 | 数据来源 | 结果 |
|------|--------|----------|------|
| 快速模式 | 18题（现有） | `data/questions-fast.js` | 16种人格类型 |
| 慢速模式 | 30题（新增） | `data/questions-slow.js` | 16种人格类型 |
| 情侣模式 | 快速或慢速各自完成 | 云数据库房间同步 | 人格类型 + 契合度分析 |

单项目，不使用分包，3模式共享全部页面逻辑。


## 二、数据流

```
首页 (index)
  ├─ 选择 [快速/慢速]  → 写 globalData.mode = 'fast'|'slow'
  │    → quiz 页（题目按 mode 加载）
  │         → result 页（显示人格类型）
  │
  └─ 选择 [情侣模式]
       ├─ 创建房间 → couple-room 页（显示4位码，轮询等待）
       └─ 加入房间 → couple-room 页（输入码，确认后进入答题）
            → quiz 页（结束后 submitResult 到云DB）
                 → couple-room 等待页（轮询对方状态）
                      → couple-result 页（双人分析）
```

页面间通信：全部通过 `getApp().globalData`，不使用 URL 参数（与现有架构一致）。

```js
// globalData 新增字段
{
  mode: 'fast' | 'slow',          // 答题模式
  coupleMode: true | false,        // 是否情侣模式
  roomCode: '1234',                // 4位房间码
  roomId: 'db-doc-id',            // 云DB文档ID
  myRole: 'A' | 'B',              // 本人角色
}
```

## 三、情侣模式流程

### 3.1 房间码设计
- 4位纯数字，由云函数生成（随机1000-9999，重试避免碰撞）
- 云DB collection `rooms` 文档结构：

```json
{
  "_id": "auto",
  "code": "3847",
  "createdAt": "timestamp",
  "expiresAt": "timestamp+24h",
  "status": "waiting|both_ready",
  "playerA": {
    "openid": "...",
    "mode": "fast",
    "result": null
  },
  "playerB": {
    "openid": "...",
    "mode": "fast",
    "result": null
  }
}
```

### 3.2 状态机

```
创建方(A):  createRoom → 显示码 → 等待B加入 → 开始答题 → 提交结果 → 等待B完成 → 跳转结果
加入方(B):  输入码 → joinRoom → 开始答题 → 提交结果 → 等待A完成(如A先完成则直接跳转)
```

### 3.3 轮询策略
- couple-room 等待页：`setInterval` 每 3 秒查询云DB
- result 提交后跳转等待：同样每 3 秒轮询
- 检测到 `status === 'both_ready'` 时清除定时器，跳转 couple-result
- 页面 `onHide`/`onUnload` 时清除定时器（防内存泄漏）

### 3.4 边界情况
- 房间码不存在 / 已过期：提示用户重新获取
- 已有2人进入的房间：提示房间已满
- 一方中途退出：另一方等待超时（5分钟）后提示可重新开始

## 四、双速题库设计

### 快速模式（18题，现有不变）
每维度3题，score≥2 → H。

### 慢速模式（30题，5题/维度）

阈值：每维度5题，**score≥3 → H**（≤2 → L）。

题目按维度分布如下（每题4选项，A/B=score 0，C/D=score 1）：

#### 维度1：亲密主动性（intimacy）
1. 认识一个很喜欢的人，你会？— 等待表白(0) / 接近不说破(0) / 暗示观察(1) / 直接表达(1)
2. 约会计划通常是谁发起？— 几乎对方(0) / 对方多(0) / 我多(1) / 几乎我(1)
3. 对方几天没主动联系，你会？— 等着不打扰(0) / 试探性消息(0) / 直接问(1) / 直接打电话(1)
4. 节假日，你倾向于？— 等对方安排(0) / 发开放问题等(0) / 列选项让对方挑(1) / 直接订好惊喜(1)
5. 关系进展（同居/旅行），你会？— 等对方开口(0) / 多次暗示(0) / 趁机开口讨论(1) / 直接提议(1)

#### 维度2：情感深度（emotion）
1. 看感人电影时，你会？— 冷静思考(0) / 有触动压住(0) / 悄悄湿眼眶(1) / 直接哭出来(1)
2. 失恋后，你靠什么走出来？— 分析失败原因(0) / 专注工作转移(0) / 和朋友倾诉释放(1) / 写日记充分经历(1)
3. 爱情最重要的是？— 互相尊重稳定(0) / 价值观一致合适(0) / 心动激情情绪(1) / 灵魂深度共鸣(1)
4. 看到对方突然低落，第一反应？— 问问解决问题(0) / 陪着给空间(0) / 轻抱说"我在"(1) / 想方设法逗开心(1)
5. 两人世界，更看重？— 共同完成有意义的事(0) / 一起讨论有趣话题(0) / 单纯待在一起感受(1) / 分享深处秘密(1)

#### 维度3：边界感（boundary）
1. 恋爱后，个人时间你会？— 都想和对方在一起(0) / 偶尔需要自己时间(0) / 保留固定个人时间(1) / 私人空间不能少(1)
2. 对方提出看你的手机？— 完全没问题(0) / 奇怪但可接受(0) / 婉拒并解释(1) / 绝对不行是底线(1)
3. 什么程度愿意共享密码？— 在一起就可以(0) / 稳定后主动分享(0) / 需要时才给(1) / 不会不管关系多好(1)
4. 伴侣想加入你与老友聚会？— 当然欢迎(0) / 可以打个招呼(0) / 提前告知我安排(1) / 婉拒单独维护(1)
5. 希望伴侣了解你的程度？— 完全透明无秘密(0) / 大部分知道(0) / 主要面貌部分保留(1) / 有不分享给任何人的部分(1)

#### 维度4：安全感（security）
1. 伴侣出差一周，你会？— 每天固定打电话(0) / 希望他/她多联系(0) / 自己充实想了联系(1) / 正好做自己的事(1)
2. 伴侣和异性好友外出，感受？— 会反复想不安(0) / 希望多汇报(0) / 信任偶尔会想(1) / 完全没影响(1)
3. 理想的两人关系状态？— 每天黏在一起(0) / 主要时间在一起(0) / 各自生活+共同时间(1) / 独立生活为主(1)
4. 对于婚姻/长期承诺，态度？— 越早确定越好(0) / 自然水到渠成(0) / 不急看缘分(1) / 不一定需要形式(1)
5. 对方突然需要"冷静期"，反应？— 非常担心要知道原因(0) / 焦虑但会等(0) / 理解给空间(1) / OK我也想想(1)

#### 维度5：表达方式（expression）
1. 喜欢一个人时，更愿意？— 默默做好小事(0) / 偶尔准备惊喜(0) / 经常用语言告诉(1) / 在社交平台公开(1)
2. 朋友问你感情怎样，你会？— "还好啦"带过(0) / 说基本情况(0) / 聊一聊分享趣事(1) / 详细倾诉包括烦恼(1)
3. 面对争吵，你通常？— 安静等情绪平稳(0) / 写出来再沟通(0) / 当面说清楚(1) / 情绪上来就表达(1)
4. 希望伴侣如何表达爱？— 默默关照生活细节(0) / 需要时出现(0) / 经常说出来(1) / 对外让人感受到(1)
5. 对于纪念日，你会？— 默默记着低调庆祝(0) / 做他/她喜欢的事(0) / 精心策划认真庆祝(1) / 公开晒出来见证(1)

#### 维度6：关系目标（goal）
1. 理想的伴侣关系，更希望？— 有人陪伴不孤单(0) / 共同建立稳定家庭(0) / 互相激励共同成长(1) / 各自追目标彼此支持(1)
2. 伴侣有机会但需要异地，你会？— 希望他/她留下(0) / 不情愿但支持短期(0) / 支持不影响感情(1) / 非常鼓励各自发展(1)
3. 关系带给你最大的价值是？— 有人懂我不孤单(0) / 家庭温暖与稳定(0) / 相互成长突破自我(1) / 自由探索有伴同行(1)
4. 对于伴侣的职业发展，更希望？— 稳定工作多陪伴(0) / 适当发展兼顾家庭(0) / 全力追梦我支持(1) / 各自拼搏不互相妥协(1)
5. 五年后，希望你们的关系是？— 默契老夫老妻(0) / 有孩子完整的家(0) / 各自有成就互相引以为傲(1) / 还在不断探索挑战(1)

## 五、契合度算法

```js
// utils/compat.js

const WEIGHTS = {
  intimacy:   0.10,
  emotion:    0.15,
  boundary:   0.20,  // 冲突高发区，权重高
  security:   0.20,  // 冲突高发区，权重高
  expression: 0.10,
  goal:       0.25,  // 价值观核心，权重最高
};

// 各维度在不同配对下的契合分（0-100）
const DIM_SCORES = {
  intimacy: {
    'H+L': 95, 'L+H': 95,  // 互补：主动+被动，最理想
    'H+H': 60,              // 双主动，偶有竞争
    'L+L': 25,              // 双等待，关系难推进
  },
  emotion: {
    'H+H': 95,              // 双感性，情感共鸣深
    'L+L': 90,              // 双理性，稳定舒适
    'H+L': 45, 'L+H': 45,  // 一方觉得对方冷漠/过于感情用事
  },
  boundary: {
    'H+H': 92,              // 双独立，互相尊重空间
    'L+L': 88,              // 双融合，共生感强
    'H+L': 25, 'L+H': 25,  // 一方窒息，一方孤独（高冲突）
  },
  security: {
    'H+H': 90,              // 双自由，不互相束缚
    'L+L': 90,              // 双稳定，需求一致
    'H+L': 30, 'L+H': 30,  // 一方焦虑，一方感到压力
  },
  expression: {
    'H+H': 88,              // 双外放，沟通顺畅
    'L+L': 85,              // 双内敛，理解彼此
    'H+L': 58, 'L+H': 58,  // 有摩擦，但可磨合
  },
  goal: {
    'H+H': 98,              // 双成长，志同道合
    'L+L': 95,              // 双陪伴，温暖稳定
    'H+L': 20, 'L+H': 20,  // 根本价值冲突（最危险）
  },
};

// keyA, keyB: 6位 H/L 字符串，顺序: intimacy/emotion/boundary/security/expression/goal
function calcCompatibility(keyA, keyB) {
  const dims = ['intimacy','emotion','boundary','security','expression','goal'];
  let total = 0;
  dims.forEach((dim, i) => {
    const pair = `${keyA[i]}+${keyB[i]}`;
    total += DIM_SCORES[dim][pair] * WEIGHTS[dim];
  });
  return Math.round(total);
}

// 分数标签
function compatLabel(score) {
  if (score >= 85) return { label: '天作之合', desc: '核心价值高度一致，默契天成' };
  if (score >= 70) return { label: '默契伴侣', desc: '整体适配，小差异反而有趣' };
  if (score >= 55) return { label: '互补组合', desc: '有分歧但可互补，需要沟通' };
  if (score >= 40) return { label: '需要磨合', desc: '某些维度差距明显，需主动努力' };
  return { label: '差异显著', desc: '核心需求有冲突，挑战不小' };
}
```

### couple-result 页展示内容
1. 两人人格类型名称 + 简介（各自）
2. 总契合度大圆环进度条 + 标签
3. 6维度对比条形图（A的维度 vs B的维度，高亮最强/最弱维度）
4. 最强契合维度说明 + 最需关注维度说明
5. 一键生成分享图（Canvas 2D，与 result 页一致）

## 六、云开发配置

### 环境
使用微信云开发（CloudBase），在 `app.js` 中初始化：
```js
wx.cloud.init({ env: 'your-env-id', traceUser: true });
```

### 云函数列表
| 函数名 | 触发方式 | 功能 |
|--------|----------|------|
| `createRoom` | 调用 | 生成4位码，创建rooms文档，返回 roomId+code |
| `joinRoom` | 调用 | 验证码，写入playerB openid，返回 roomId+mode |
| `submitResult` | 调用 | 写入本人result，若双方都ready则更新status |

### 数据库集合：rooms
```
权限：仅创建者可读写自己的playerA字段，
      已加入的playerB可读写自己的playerB字段，
      双方均可读status字段
```

## 九、未覆盖的边界情况处理

| 情况 | 处理 |
|------|------|
| 加入不存在的房间码 | 提示"房间不存在或已过期" |
| 房间已满（已有B） | 提示"房间已被加入" |
| 等待超过5分钟对方无响应 | 提示"对方可能已离开，可重新开始" |
| 页面onUnload时 | 清除所有轮询定时器 |
| result页globalData为空 | onLoad检测到空则跳回首页（现有逻辑不变）|
